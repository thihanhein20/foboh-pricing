⏺ Here's what I found, grouped by severity:

  ---
  Logic Bugs

  1. adjustmentType: "dynamic" silently falls through to percentage math
  (calculateNewPrice, line 27)

  The AdjustmentType union is "fixed" | "dynamic", but the check is only === 
  "fixed". Anything that isn't "fixed" — including "dynamic" — is silently
  treated as percentage. If "dynamic" was meant to be something other than a
  percentage, this is a silent mis-calculation.

  2. profile.category matched against product.subCategory
  (profileMatchesProduct, line 78)

  return profile.category === product.subCategory;

  Product has no category field — only subCategory and segment. If the UI or
  data layer populates profile.category from something other than a product's
  subCategory value, this comparison is silently always false and no
  category-scoped profile ever matches.

  3. Negative adjustmentValue inverts direction (calculateNewPrice, lines 27–36)

  No guard against adjustmentValue < 0. A profile with adjustmentDirection: 
  "decrease" and adjustmentValue: -10 will actually increase the price
  (basePrice - (-10)). Math.max(0, ...) only catches the final floor, not this
  semantic inversion.

  ---
  Edge Cases

  4. "Greatest saving" tiebreaker when all tied profiles are increases

  const savingA = product.basePrice - calculateNewPrice(product.basePrice, a);

  For increase profiles, savingA is negative. The sort still produces a
  deterministic result (smallest increase wins = cheapest price for customer),
  which may be the intended behaviour — but the word "saving" in the variable
  name is misleading and the rule comment on line 11 doesn't clarify how this
  should behave for increase-only ties.

  5. Missing required fields on "specific"-scope profiles silently never match

  In profileMatchesCustomer (line 58): if profile.customerScope === "specific"
  but profile.customerId is undefined, it evaluates undefined === customerId →
  false. Same for customerGroup on group-scope profiles. These profiles are
  silently ignored rather than producing a warning or error, which can make
  misconfigured profiles hard to diagnose.

  ---
  Type Safety

  6. appliedProfileId: "none" is a magic string (line 109)

  PriceResolution.appliedProfileId is typed as string. Returning the sentinel
  "none" when no profile applies forces callers to know this convention. string 
  | null (or a separate appliedProfile: null field) would be more type-safe.

  ---
  Minor / Code Quality

  7. .sort() mutates matchingProfiles in place (line 115)

  const sorted: PricingProfile[] = matchingProfiles.sort(...);

  Array.sort is in-place. sorted is an alias, not a copy. Not a bug here since
  matchingProfiles isn't used after this point, but it's misleading —
  [...matchingProfiles].sort(...) would be safer.

  8. Redundant customers.find() / products.find() in helpers (lines 52, 71)

  resolvePrice already guards against missing customer/product at line 95 before
  calling the filter. Both helper functions then do the same lookup again for
  every profile in the list. For large datasets this is O(n×m) when O(n+m) is
  achievable.

  ---
  The highest-priority ones to look at are #2 (category matching could be
  silently broken depending on your data), #1 (the "dynamic" type handling), and
  #5 (misconfigured profiles being invisible).

❯ What edge cases am I not handling in my resolver? Think about: what if        
  adjustmentValue is 0, what if two profiles have identical scores AND          
  identical savings, what if productIds is an empty array, what if a product is 
  deleted after a profile is created referencing it.                            

⏺ Working from the code I already read, here's what happens in each case:

  ---
  adjustmentValue: 0

  Works mathematically — basePrice ± 0 = basePrice. But the profile is still
  selected as the "winner" and shown as applied in the resolution. The output
  says profile X was applied with a reason string saying it won, while the price
  is completely unchanged. A misconfigured zero-value profile silently shadows
  a legitimate discount profile that ranked lower. No way for the caller to know
  the winning profile had no effect.

  ---
  Identical scores AND identical savings

  The sort comparator returns 0, so the winner falls back to JavaScript's sort
  stability — whichever profile appeared first in the input profiles array
  survives. Modern Node.js sort is stable (V8 TimSort), so this is deterministic
  for a given call, but the result depends entirely on the order the profiles
  array was loaded/passed in. If that order changes (e.g. database re-ordering
  after an upsert, or pagination), the winner changes silently.

  The other problem: the reason field still says "most specific won" even when
  it was effectively a coin flip. No way for the caller to detect a tie
  occurred.

  ---
  productIds: [] (empty array)

  return profile.productIds?.includes(productId) ?? false;

  [].includes(productId) → false. A "specific" scope profile with an empty
  productIds array will never match any product. It's a permanently dead profile
  with no error or warning. Same applies if productIds is missing entirely
  (undefined → false via ??). Both silently no-op.

  ---
  Product deleted after profile references it

  Four distinct sub-cases:

  1. resolvePrice called for the deleted product — products.find(p => p.id === 
  productId) returns undefined, function returns null. The caller can't
  distinguish this from "customer not found" — both return null. There's no way
  to tell whether null means "no such product", "no such customer", or some
  other missing entity.
  2. Zombie profiles in profileMatchesProduct — profiles with productIds: 
  ["deleted-id"] just silently never match anything. They remain in the system
  claiming to target a product that no longer exists, invisible to operators.
  3. profileMatchesCustomer redundant lookup — as noted in the earlier audit,
  the helpers do their own customers.find() / products.find() per profile. For
  deleted products this doesn't cause a bug (the guard at line 95 fires first),
  but the helpers aren't actually reachable via the deleted-product path — the
  dead code makes the flow harder to reason about.
  4. ID reuse — if the deleted product's ID is later assigned to a new product,
  any zombie profiles that referenced the old product will suddenly start
  matching the new one. Depending on how IDs are generated this may or may not
  be realistic, but the resolver has no tombstone or validity check to prevent
  it.
